SIMION Quadrupole Example

Purpose

This example models an RF quadrupole mass filter with the RF potentials applied via a user workbench program. This is a relatively complete model with entrance and exit regions modeled via 3D arrays. General background and theory on the quadrupole mass filter are available separately in the references at bottom.

Files in this demo:

  • README.html - Description of example
  • quad.iob - workbench of main example
    • quad.pa# - potential array of center rod 2D planar region. Generated by quad.gem.
    • quadout.pa# - potential array of detector 3D region. Generated by quadout.gem.
    • quadin.pa# - potential array of entrance 3D region. Generated by quadin.gem.
    • quad_include.gem - GEM file for entire geometry. This is included by the other GEM files.
    • quad.fly2 - particle definitions, randomized at entrance
    • quad.lua - workbench user program, applying DC and RF voltages from adjustable variables.
  • quad_monolithic.iob - similar to quad.iob but using single potential array (simpler but uses more RAM)
    • quad_monolithic.pa# - potential array of entire system. Generated by quad_monolithic.gem
    • quad_monolithic.fly2 - same as quad.fly2
    • quad_monolithic.lua - identical in behavior to quad.lua, by importation

Description

This demo couples potential arrays (PAs) and user programs to model a complex problem--a full quad including entrance and exit asymmetrical regions. RAM requirements are around 20 MB total. This demo can serve as a starting point for modeling a specific quad.

Modeling

Particle definitions: Ions are generated randomly in energy, cone angle, radial offset, and phase using a FLY2 file (quad.fly2), which is viewable and editable from "Particles|Define..." on the SIMION View screen. The ion time of births are uniformly randomly distributed between 0 µsec and 0.909 µsec, which is covers one full RF cycle when the rods are operated at 1.1MHz, thereby ensuring that all acceptance phase angles are represented.

RF voltages: RF voltages are applied to the quadrupole rods with a user workbench program, quad.lua. More detailed discussion of the model and its parameters is contained in the comments in the quad.lua file.

PA's: Although the system can most simply be modeled as a single potential array (as done in quad_monolithic.iob), it is possible to model this is a more RAM efficient way by splitting the system into multiple (three) potential arrays. Some care is needed when modeling systems with multiple PAs since the Laplace equation is solved independently in each array (see Doc(Multiple PAs), but this is a good example of how to handle it. Note that this system has almost 2D planar symmetry on the long rods, but the entrance and exit lenses break that symmetry, unfortunately preventing us from simply modeling the entire system as a 2D potential array. The obvious (simplest and least error prone) solution is to model the system with a 3D potential array, but this could require a fair amount of RAM to achieve high accuracy. It would be desirable, therefore, to cut the system into three potential arrays: a 3D entrance lens, a 3D exit lens, and the long rods with 2D planar symmetry (modeled at high resolution). The question is where to make the cuts. To determine this, the system was first modeled as one big (but course) 3D potential array (e.g. like quad_monolithic.gem/pa#). The result provided an indication of how far the fields penetrate through the entrance and exit lens and into the rod section. The system was then cut into three potential arrays at locations far enough away from the entrance and exit such that the fields are largely 2D symmetric along the length of the rods. All three potential arrays were added to a workbench. The length of the 2D center array section was adjusted via the nz parameter in the instance edit field (PAs | Positioning | nz use). This also means that it is easy to change the quad's length!--just change the nz value and shift the quad and quadout instances so that all three instances connect to form a complete quadrupole.

Circular rod radius: An ideal quadrupole field has equipotential surfaces in the shape of hyperbolas, so in theory the desired field could be formed with perfectly shaped hyperbolic rods of constant potential. Often we instead use circular rods of constant potential, which are easier to machine. As noted in Dawson1997, p. 117, given an ideal quadropole field with a half radius between rods of r0, this field can be approximated with circular rods of the same potential and half radius between rods provided the rods themselves have a radius of about r = 1.148 r0, which minimizes sixth-order error (DaytonShoemaker1954). Denison1971 alternately suggested minimizing sixth-order error by using r = 1.1468 r0 in conjunction with a ground cage of radius R=3.54 r0.

Surface enhancement: This example now uses surface enhancement to reduce the errors in electric field calculation accuracy due to curved rod electrodes. This was enabled by the surface=fractional in the GEM files. Surface enhancement is quite useful because in this example there is only about 20 PA grid units (gu) along the inter-rod radius (as most readily seen in quad.pa#), there are ± 0.5 gu surface errors without surface enhancement, and these long rods do most of the focusing. Another way to improve accuracy is to increase the number of grid points, particularly for quad.pa# (which as a 2D PA and doesn't consume that much RAM).

Usage

How to run quad demo:

  • 1. View quad.iob. (Click View on main screen and select the quad.iob file.)
  • 2. If this is your first time using the example, you will be asked to generated the PA's from GEM filse and refine (confirm that and press Refine).
  • 3. Select trajectory quality = 0 and disable Rerun. (In Particles tab, set TQual to 0 and disable Rerun.)
  • 4. Fly ions separately as lines. (In Particles tab, disable Dots. Click Fly'm.)
  • 5. View ions in side, top, and end 2D views. Cut top off of quad and use isometric view to look inside. (Try XY, ZY, XZ, and 3D Iso views. For 2D/3D zooming and cutting options, see details in printed SIMION manual, or skip this.)
  • 6. Now fly quad1.fly2 file with the parameters above. (In Particles tab, click Define..., click Load, select quad1.fly2, and press OK. Refly by pressing Fly'm.) Notice how the lighter ions impact one pole pair while the heavier ions impact the other pole pair.
  • 7. Examine effect of changing quad axis potential. You have control of the quad axis potential via an adjustable variable. Examine the relationship between it and how the ions fly through the quad. (In Variables tab, change _quad_axis_voltage from -8 to 0 and press enter. Click Fly'm to refly.)
  • 8. Vary the ion's cone angle to 15 degrees. (In Variables tab, enable Show All, change cone_angle_off_vel_axis from 5 to 15, and press enter. Click Fly'm to refly.) Notice the increase in ion loses (many are outside of acceptance ellipses).
  • 9. Fly the ions Grouped as Dots at high speed with Rerun active. (In Particles tab, enable Grouped, Dots, and Rerun. Move slider below Dots to far right for high speed.)
  • 10. Cut the quad in half via 3D zoom and look at the PE surfaces undulate. (In XY view, drag box around center section--vertically long and horizontally thin. Click +Z3D button. Press ZY button. Press PE button.) Note that only portions of quad that have ions in them undulate. Why? Be sure you cut the quad in all three planes and explore with the PE view options. (When done viewing, click -Z3D and unpress PE.) Faster PE surface updates occur with drawing quality at zero and GStep at 6 or so (Display Qual panel and PE Contours tab, GStep).
  • 11. Experiment on the effects of tuning, ion initial conditions, etc. This model provides a lot of insight into why quads work like they do if you spend some time with it.
  • 12. You can modify the user programs and PAs to reflect a real world quad. This demo serves as a good starting point for quad models.
  • 13. The quad_monolithic.iob example can be tried out for comparison. This is basically identical but uses a single PA (simpler but uses more RAM).

Some Additional Notes

Finding the Mathieu Constants q_max and a_max

As noted in Abramowitz and Stegun Handbook of Mathematical Functions[*] referenced from Wikipedia page on the Mathieu function, power series expansions of the characteristic values are
a_0(q) = (-1/2)*q^2 + (7/128)*q^4 + (-29/2304)q^6 + (68687/18874368)*q^8 + ...

b_1(q) = 1 + (-1)*q + (-1/8)*q^2 + (1/64)*q^3 + (-1/1536)*q^4 + (-11/36864)*q^5 + (49/589824)*q^6 + (-55/9437184)*q^7 + (-83/35389440)*q^8 + ...

The curves -a_0(q) and b_1(q) on the a-q plane define the limits of stability. The maximum a (a_max) in this region is reached at the intersection point (q_max, a_max) of these two curves. Solving for the root of b_1(q) + (-1)*a_0(q) = 0 (numerically) gives q_max ~ 0.70601. Plugging that into b_1(q) gives a_max ~ 0.23698.

Notice that the last term of a_0(q) is on the order of 2*10^-4 and decreases by a factor of ~10 for each term. So, close to five digits of accuracy should be right, as consistent with the numbers used in quad.lua.
